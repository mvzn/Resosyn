## What This Is

Resosyn is a JUCE audio plugin (C++17). Currently a stub — `processBlock()` and state persistence are empty. The project targets VST3 and Standalone on Linux.

## Architecture

- `PluginProcessor` handles all audio processing and owns DSP objects. UI must not contain DSP logic.
- `PluginEditor` handles all drawing and user interaction. It must be deletable without affecting audio.
- Use `AudioProcessorValueTreeState` (APVTS) for all user-facing parameters. Define the parameter layout in a single dedicated function.
- Each logical DSP component (filter, delay line, envelope follower, etc.) gets its own class. `PluginProcessor` composes them — it should not contain raw DSP math inline.
- Always implement `getStateInformation()` / `setStateInformation()` so DAW sessions restore plugin state correctly.

## DSP Constraints

- No memory allocations in `processBlock()` — pre-allocate in `prepareToPlay()`.
- No locks or blocking operations on the audio thread.
- Use `juce::SmoothedValue` for all audio-rate parameter changes to prevent zipper noise.
- Reach for `juce::dsp` module processors before writing custom DSP.

## Code Style

- `camelCase` for variables and methods, `PascalCase` for classes.
- Always use the `juce::` namespace prefix explicitly — never `using namespace juce;`.
- Prefer RAII — acquire in constructors, release in destructors.
- No raw pointers — use `std::unique_ptr`, `std::shared_ptr`, or JUCE equivalents.

## Coding Principles

- **Single responsibility.** Every function does one thing. If you need "and" to describe it, split it.
- **Small functions.** If a function exceeds ~30 lines, extract sub-operations with descriptive names.
- **No god classes.** If a class is accumulating unrelated methods, decompose it.
- **Composition over deep inheritance.** One or two levels is fine — five is a design smell.
- **Descriptive names over comments.** A function called `calculateDelayInSamples()` needs no comment.
- **Comment why, not what.** Explain the reasoning behind non-obvious decisions, not what the code does.
- **Fail early and clearly.** Validate inputs at function boundaries — don't let bad state propagate silently.

## What NOT To Do

- Don't put DSP logic in the editor/UI code.
- Don't use `juce::var` or dynamic typing where static types work.
- Don't add third-party dependencies without confirming first.
- Don't over-engineer — keep plugin architecture simple and flat. No deep inheritance hierarchies.
- Don't add features not specified in the active PRD.
